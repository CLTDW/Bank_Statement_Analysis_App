# 银行流水分析程序

本项目包含两个核心程序，构成完整的银行流水数据处理Pipeline：

1. **银行流水汇总程序** - 批量汇总和合并多个银行流水文件
2. **银行流水清洗程序** - 数据清洗、校验和标准化处理

### 核心类详解

#### 1. BankStatementAggregator（主控制器）

**职责**：程序核心控制器，负责协调整个流水处理流程

**核心方法**：
- `run()` - 主入口，协调各阶段执行
- `_detect_system_config()` - 检测硬件配置（CPU核心数、内存大小）
- `_generate_resource_plan()` - 生成最优资源分配方案
- `_calculate_optimal_workers()` - 动态计算并行度
- `_backup_original_files()` - 自动备份原始文件

**设计亮点**：
- **自适应资源调度**：根据硬件配置动态调整并行度和批处理大小
- **分层错误处理**：处理失败的文件自动移至待复核文件夹
- **完整的审计日志**：记录每一步操作，便于问题排查

#### 2. FileProcessor（文件处理器）

**职责**：处理不同类型的银行流水文件

**核心方法**：
- `process_single_file()` - 处理单个文件
- `_process_large_file()` - 分块处理大文件（>10MB）
- `_process_large_multi_account_file()` - 处理多账户大文件
- `_calculate_header_score()` - 计算表头匹配度得分
- `_check_memory_usage()` - 内存监控和紧急清理

**设计亮点**：
- **多级缓存机制**：Excel文件缓存、编码缓存、分隔符缓存、表头得分缓存
- **智能缓存清理**：基于LRU策略和内存阈值的自动清理
- **大文件分块处理**：避免内存溢出，支持任意大小文件

#### 3. MappingManager（映射管理器）

**职责**：管理表头映射和关键词匹配规则

**核心数据结构**：
- `header_mapping` - 精确表头映射（如"交易时间" → "交易时间"）
- `keyword_mapping` - 关键词映射（如包含"金额"的列映射到"交易金额"）
- `expected_columns` - 预期输出列（从模板文件加载）
- `surnames` - 姓氏库（用于识别户主名称）

**设计亮点**：
- **多层次映射策略**：精确匹配优先，关键词匹配兜底
- **可配置规则**：所有映射规则通过YAML配置文件管理
- **去重机制**：自动排查重复映射，保证映射唯一性

#### 4. MultiAccountHandler（多账户处理器）

**职责**：处理包含多个账户的流水文件

**核心能力**：
- 自动识别文件中是否包含多个账户的表头
- 根据户主名称拆分数据
- 支持无户名单账户处理模式

### 配置文件详解

| 配置项 | 类型 | 说明 | 默认值 |
|-------|------|------|--------|
| `base_dir` | string | 流水文件目录 | E:\流水分析 |
| `output_file` | string | 输出文件路径 | 银行流水文件汇总结果.xlsx |

| `large_file_threshold` | int | 大文件阈值（字节） | 10MB |
| `parallel.enabled` | bool | 是否启用并行处理 | true |
| `parallel.max_workers` | int | 最大工作线程数 | 32 |

**mappings.yaml 配置结构**：

```yaml
# 表头精确映射
header_mapping:
  '交易日期': '交易日期'
  '发生日期': '交易日期'
  '日期': '交易日期'
  
# 关键词映射（用于模糊匹配）
keyword_mapping:
  '交易日期': ['日期', '发生日期', '交易日']
  '交易金额': ['金额', '发生额', '收支']
  
# 匹配规则
matching_rules:
  exclusion_rules:  # 排除规则
  surnames:         # 姓氏库
    single: []      # 单姓
    compound: []    # 复姓
```

### 核心类详解

#### 1. DataValidator（数据校验器）

**职责**：根据规则校验数据有效性

**核心校验方法**：
- `validate_field_completeness()` - 检查必需字段组合
- `validate_account_holder()` - 校验户主名称（长度、汉字占比、姓氏匹配）
- `validate_bank()` - 校验开户行（必须包含银行关键词）
- `validate_account_number()` - 校验账号/卡号（长度、数字占比）
- `validate_date_time()` - 校验日期时间（支持多种格式）
- `validate_amount()` - 校验金额（必须为数字）
- `validate_batch()` - 批量校验数据

**校验规则**：
| 字段 | 校验规则 |
|-----|---------|
| 户主名称 | 长度2-30字符，汉字占比≥60%，不得为纯数字 |
| 开户行 | 长度2-30字符，汉字占比≥90%，必须包含银行关键词 |
| 银行卡类别 | 长度2-15字符，不得包含数字 |
| 账号/卡号 | 长度8-30位，数字占比≥75% |
| 交易日期 | 必须为有效日期格式 |
| 交易金额 | 必须为数字 |

#### 2. DataCleaner（数据清洗器）

**职责**：执行数据清洗操作

**核心清洗方法**：
- `remove_duplicate_and_empty_rows()` - 删除重复行和空白行
- `calculate_transaction_amount()` - 计算交易金额（贷方-借方）
- `unify_debit_credit()` - 统一借贷标志为"收入"/"支出"
- `process_tenpay_data()` - 处理Tenpay数据（金额/100）
- `extract_information()` - 从文件名提取账户信息
- `match_account_holder()` - 匹配户主名称
- `remove_zero_amount_rows()` - 删除金额为0的行

**清洗流程**：
```
原始数据 → 过滤全空行 → 删除重复行 → 计算交易金额 → 统一借贷标志
        → 处理特殊数据 → 提取账户信息 → 匹配户主名称 → 删除无效行
```

#### 3. DateTimeProcessor（日期时间处理器）

**职责**：处理日期时间字段

**核心方法**：
- `normalize_datetime_columns()` - 标准化日期时间格式
- `merge_datetime_columns()` - 合并日期和时间列

**支持的日期格式**：
- `YYYY-MM-DD HH:MM:SS`
- `YYYY/MM/DD`
- `YYYY年MM月DD日`
- `YYYYMMDD`
- Excel序列号

#### 4. HardwareMonitor（硬件监控器）

**职责**：监控系统资源使用情况

**核心能力**：
- 动态计算资源使用阈值（根据CPU核心数和内存大小）
- 实时监控CPU和内存使用率
- 自动限流避免资源耗尽

### 推荐使用流程

```

```

**场景二：大规模/质量参差不齐的数据**

## 技术亮点

### 智能资源管理
- **动态并行度**：根据CPU核心数和内存使用情况动态调整并行线程数
- **智能批处理**：根据文件大小自动选择分块大小
- **紧急内存清理**：内存使用率超过阈值时自动清理缓存

### 多账户识别
- **自动检测**：识别文件中是否包含多个账户的表头
- **智能拆分**：根据户主名称自动拆分数据
- **无户名处理**：支持处理没有户主名称的文件

### 数据质量保障
- **多层次校验**：字段完整性、格式、业务规则校验
- **自动备份**：处理前自动备份所有原始文件
- **异常处理**：失败文件自动移至待复核文件夹

### 可扩展性设计
- **配置驱动**：所有规则通过配置文件管理，无需修改代码
- **模块化架构**：各模块职责清晰，易于扩展
- **支持多格式**：Excel、CSV、PDF、TXT等多种文件格式

## 注意事项

1. **文件格式**：支持 `.xlsx`, `.xls`, `.csv`, `.txt`, `.pdf` 格式
2. **文件大小**：建议单个Excel文件不超过50MB，大数据量请分批处理
3. **编码问题**：程序会自动检测文件编码，但建议使用UTF-8编码
4. **模板文件**：确保模板文件存在且格式正确
5. **日志查看**：处理过程会生成详细日志，便于问题排查
6. **备份机制**：程序会自动备份原始文件到"原始文件备份"目录
