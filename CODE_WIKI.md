# 银行流水汇总程序 Code Wiki

## 项目概述

银行流水汇总程序是一款专为财务人员设计的自动化数据处理工具，其核心功能是将来自不同银行、不同格式的流水文件进行统一汇总和标准化处理。该程序能够自动识别各类银行流水的表头格式，提取关键交易信息，并生成符合统一规范的汇总报表，极大地提升了财务对账和数据分析的工作效率。

本程序采用模块化架构设计，具有良好的可扩展性和可维护性。通过配置文件驱动的方式，用户可以灵活地添加新的银行格式支持或调整数据处理规则，无需修改核心代码。程序内置了智能的资源监控系统，能够根据计算机的硬件配置动态调整并行处理策略，在保证处理速度的同时避免系统资源过载。此外，程序还提供了完善的日志记录和错误处理机制，支持自动备份原始文件，确保数据安全性和可追溯性。

### 模块职责划分

**核心业务层（src/core/）** 承担了程序的主要业务逻辑。`bank_statement_aggregator.py` 作为程序的核心调度器，负责协调各个模块的工作，控制整个处理流程的执行顺序，包括文件收集、并行处理、结果合并和报告生成等关键环节。`config_manager.py` 专注于配置的加载、验证和访问，提供统一的配置访问接口。`mapping_manager.py` 负责维护表头映射和关键词映射的规则，支持从配置文件动态加载映射关系。`file_processor.py` 是文件处理的执行者，根据文件类型调用相应的解析器，并负责表头识别和数据提取。

**工具层（src/utils/）** 提供了各种辅助功能。`file_utils.py` 封装了文件操作的常用功能，包括文件收集、格式检测和临时文件管理等。`text_utils.py` 提供了文本处理相关的工具函数，如日期解析、金额格式化等。`data_utils.py` 包含数据转换和验证的工具函数。`multi_account_handler.py` 用于处理多账户场景下的数据合并。

**文件处理层（src/file_processing/）** 负责处理特定文件格式。`pdf_processing.py` 实现了从PDF文件中提取表格数据的功能，支持多种PDF格式的解析。

## 核心模块详解

### 配置管理模块（config_manager.py）

`ConfigManager` 类是系统的配置管理中枢，采用单例模式的思想设计，确保配置信息在程序运行期间保持一致。该模块的核心职责包括配置文件的加载、配置项的读取和修改，以及配置路径的管理。

在初始化阶段，类构造函数接收配置文件路径参数，默认为 `config.yaml`。`load_config` 方法使用 `PyYAML` 库解析配置文件，将配置内容存储在内存字典中。如果配置文件不存在或格式错误，模块会抛出明确的异常信息，帮助用户快速定位配置问题。配置加载成功后，所有配置项都可以通过 `get` 方法按需获取，支持指定默认值以应对配置项缺失的情况。

路径管理是配置模块的重要功能之一。该模块维护了一套完整的路径获取方法，包括 `get_base_dir` 获取基础工作目录、`get_template_file` 获取模板文件路径、`get_output_file` 获取输出文件路径、`get_log_file` 获取日志文件路径，以及 `get_report_file` 获取报告文件路径。这些方法统一使用 `pathlib.Path` 对象处理路径，确保跨平台的路径兼容性。模板文件路径默认从程序根目录查找，输出文件则保存到基础目录下。

配置模块还提供了配置更新和保存的功能。`update` 方法允许在运行时修改配置项，`save` 方法将当前配置写回文件，便于保存用户的配置修改。这些功能为程序提供了灵活的配置管理能力，用户可以根据实际需求动态调整程序行为。

### 映射管理模块（mapping_manager.py）

`MappingManager` 类负责管理系统中所有与数据映射相关的规则，包括表头映射、关键词映射、排除规则和姓氏库等。该模块的设计目标是支持多种银行流水格式的自动识别和标准化转换。

表头映射是模块的核心功能之一。由于不同银行的流水文件使用不同的列名来表示相同的字段信息（如“交易金额”在某些银行称为“发生额”，在另一些银行称为“金额”），映射管理器需要维护一套完整的映射规则，将各种原始列名映射到统一的标准列名。`header_mapping` 字典存储了从原始列名到标准列名的映射关系，`keyword_mapping` 字典则存储了用于识别各列的关键词列表，两者配合使用可以准确识别并转换表头。

映射管理器支持动态加载配置。`_load_mappings_config` 方法从 `mappings.yaml` 文件读取映射配置，包括表头映射、关键词映射和匹配规则。`_load_expected_columns` 方法从模板文件读取预期的输出列结构，确保汇总结果符合用户要求的格式。如果模板文件不可用，模块会使用配置文件中的备用列定义或硬编码的默认列定义。

排除规则用于在数据处理过程中过滤无效或冗余的信息。通过 `_load_exclusion_rules` 方法加载的排除规则可以定义哪些关键词组合应该被忽略，避免误匹配。姓氏库则用于姓名识别功能，包含常见的单姓和复姓列表，支持在流水中提取交易对手的姓名信息。

### 主程序模块（bank_statement_aggregator.py）

`BankStatementAggregator` 类是程序的核心编排器，协调整个银行流水汇总流程的执行。该类的设计遵循了清晰的关注点分离原则，将流程控制、资源管理和业务逻辑解耦到不同的组件中。

初始化阶段，主程序会依次完成以下准备工作：加载配置信息、初始化日志系统、检测系统硬件配置、创建资源监控器、生成最优资源配置方案，以及初始化映射管理器和文件处理器。系统配置检测包括CPU核心数、内存大小和磁盘速度等指标，这些信息用于后续的资源分配决策。资源监控器会实时跟踪CPU和内存使用情况，当资源紧张时自动调整处理策略。

处理流程从 `run` 方法开始执行。整个流程分为七个主要阶段：基础目录验证、用户输入获取、文件收集、原始文件备份、文件并行处理、结果合并保存和报告生成。在文件收集阶段，程序扫描指定目录及其子目录，收集所有支持格式的文件（Excel、CSV、TXT、PDF等），并根据排除规则过滤掉不需要处理的文件。

并行处理是程序的重要特性。`_calculate_optimal_workers` 方法根据当前系统资源状态和文件数量动态计算最优的并行度。当系统资源充足时，程序会使用多线程并发处理多个文件，显著提升处理效率；当资源紧张时，程序会自动降级为串行处理，避免系统过载。处理结果通过 `process_single_file` 方法返回，每个文件处理完成后会添加“数据来源”列，记录原始文件的完整路径。

结果保存阶段负责将所有处理结果合并为最终的汇总文件。程序首先合并所有DataFrame，然后根据预期列定义进行列映射和排序，最后使用openpyxl库将结果保存为Excel文件。对于数据量较大的情况（超过30万行），程序会自动分批保存，每批20万行。此外，程序还会将结果同时保存到TEST文件夹，便于用户查看和验证。

资源管理是主程序模块的重要组成部分。`ResourceMonitor` 类封装了系统资源监控的功能，通过psutil库获取CPU和内存使用率。当资源使用超过90%的阈值时，监控器会暂停任务提交，等待资源释放。`_generate_resource_plan` 方法根据系统配置生成最优的资源分配方案，包括并行线程数、批处理大小、内存块大小等参数，这些参数会影响后续文件处理的性能和稳定性。

### 文件处理器（file_processor.py）

`FileProcessor` 类负责具体文件的读取和解析工作，支持多种文件格式的自动识别和处理。该模块采用了策略模式的设计思想，为不同格式的文件提供统一的处理接口，同时保留了各格式特定的解析逻辑。

对于Excel文件，处理器使用pandas的 `read_excel` 方法读取，支持 `.xlsx`、`.xls` 和 `.et`（WPS表格格式）等格式。读取时会尝试多个sheet页，合并所有sheet的数据。CSV和TXT文件则需要处理多种编码格式（UTF-8、GBK、GB2312等）和分隔符（逗号、制表符、分号等），处理器会尝试验证并选择最合适的解析参数。PDF文件的处理最为复杂，需要使用pdfplumber库提取表格数据，解析过程中会尝试多种表格检测策略。

表头识别是文件处理的关键步骤。处理器会扫描文件的前若干行（默认15行），使用关键词匹配的方式识别各列的含义。识别结果会与映射管理器中定义的预期列进行比对，建立列名映射关系。对于无法识别的列，处理器会尝试通过列位置或列内容特征进行推断，必要时保留原始列名。

数据清洗和标准化是处理流程的重要环节。处理器会对金额字段进行格式统一，包括去除货币符号、处理负数格式（如括号表示负数）、统一小数位数等。日期字段会被转换为标准格式，支持多种输入格式的自动解析。借贷标志会被统一为“收”和“支”两种标识，便于后续的统计分析。

### 工具函数

程序中的工具函数按照功能分为以下几类：

**文件工具函数（file_utils.py）**：

`collect_all_supported_files` 函数用于递归扫描指定目录，收集所有支持格式的文件。它接收基础目录路径和配置字典作为参数，返回符合条件的文件路径列表。函数会自动排除配置中指定的文件和临时文件模式，确保只处理有效的银行流水文件。

**文本工具函数（text_utils.py）**：

`parse_date` 函数负责解析各种格式的日期字符串，将其转换为标准的 `YYYY-MM-DD` 格式。函数支持中文日期格式（如"2024年1月1日"）、标准日期格式（如"2024-01-01"）以及多种分隔符格式的自动识别。

`parse_amount` 函数处理金额字符串的标准化。它能够去除货币符号（如"￥"）、千分位分隔符（如"，"），并处理括号表示的负数格式（如"(100)"转为"-100"），最终返回标准化的浮点数格式。

**数据工具函数（data_utils.py）**：

`validate_dataframe` 函数用于验证处理后的DataFrame是否符合预期要求。它检查必要的列是否存在、数据类型是否正确、是否有空值等，并返回验证结果。

`merge_dataframes` 函数负责合并多个DataFrame对象，处理列名不一致的情况，将数据整合到统一的列结构中。

## 项目配置

### 主配置文件（config.yaml）

主配置文件采用YAML格式，包含程序运行的各项参数配置。配置分为以下几个部分：

**基础配置** 部分定义了程序运行的基本路径和文件名，包括基础工作目录 `base_dir`、表头映射表文件 `mapping_file`、流水汇总模板文件 `template_file`、输出结果文件 `output_file`、日志文件 `log_file` 和报告文件 `report_file`。日志级别 `log_level` 支持 DEBUG、INFO、WARN、ERROR 四个等级，默认使用 DEBUG 级别以便于问题排查。

**排除配置** 部分定义了需要排除的文件和文件模式。`excluded_files` 列出具体的文件名，如模板文件和输出文件本身；`excluded_patterns` 使用通配符模式排除临时文件，如 `~$*.xlsx` 匹配Excel临时文件。

**字段类型配置** 部分定义了各类字段的识别关键词。`field_types` 字典将字段分为文本、日期、金额和借贷标志四类，每类包含多个可能的列名关键词。`debit_credit_mapping` 专门定义借贷标志的统一映射，将各种表示收入的关键词映射为"收"，表示支出的映射为"支"。

**银行BIN码库** 是用于识别银行卡所属银行的核心配置。`bank_bin_database` 字典以银行为键，以BIN码列表为值，通过卡号前6位数字匹配可以快速识别银行卡的发卡银行。

**日期和金额配置** 部分定义了日期格式的优先级列表和金额处理的正则替换规则，确保各种格式的日期和金额都能被正确解析。

### 映射配置文件（mappings.yaml）

映射配置文件位于 `src/config/` 目录下，包含以下核心配置：

`header_mapping` 字典定义了从原始表头到标准表头的映射关系，这是表头识别的核心依据。配置中应尽可能包含各银行流水常见的表头变体。

`keyword_mapping` 字典定义了识别各列所需的关键词列表。与表头映射不同，关键词匹配可以处理表头不完整或格式不规范的情况。

`matching_rules` 包含匹配规则的详细配置，如 `exclusion_rules` 定义需要排除的关键词组合，`surnames` 包含用于姓名识别的姓氏库。

## 运行方式

### 基本运行

程序支持命令行参数运行和配置文件运行两种方式。直接运行是最简单的方式：

```bash
python bank_statement_aggregator.py
```

这种方式使用默认的 `config.yaml` 配置文件，程序会提示用户输入汇总对象名称，然后开始处理。

### 命令行参数

程序支持多个命令行参数，用于灵活控制运行行为：

```bash
python bank_statement_aggregator.py -c <配置文件路径> -d <工作目录> -o <输出文件> --disable-parallel
```

各参数说明如下：`-c` 或 `--config` 指定配置文件路径，默认为 `config.yaml`；`-d` 或 `--directory` 指定工作目录，覆盖配置文件中的 `base_dir` 设置；`-o` 或 `--output` 指定输出文件路径，覆盖配置文件中的 `output_file` 设置；`--disable-parallel` 禁用并行处理，改为串行处理模式。

### 环境准备

运行程序前需要确保以下环境条件满足：

Python版本要求3.7以上，推荐使用3.8或更高版本以获得更好的性能支持。所有依赖包需要通过pip安装：

```bash
pip install -r requirements.txt
```

工作目录（`base_dir`）需要预先创建并放置待处理的银行流水文件。模板文件 `流水汇总模板.xlsx` 需要放置在程序根目录，包含预期的输出列结构。

### 运行流程

程序启动后会依次执行以下步骤：

首先进行初始化校验，检查基础目录和模板文件是否存在。然后提示用户输入汇总对象名称，多个名称用逗号或顿号分隔。接着收集目录中所有支持格式的文件，自动排除临时文件和指定排除的文件。

如果启用了并行处理且文件数量较多，程序会根据系统资源情况自动选择最优的并行度。处理过程中会实时显示进度条，处理完成后会记录成功和失败的文件信息。

处理完成后，程序自动备份原始文件到带时间戳的目录，并将结果保存到指定的输出文件。最后生成处理报告，包含处理统计和失败文件列表。

### 输出结果

程序运行后会生成以下文件：

## 数据流与处理逻辑

### 整体数据流

数据在程序中的流动遵循以下路径：首先用户指定的目录中的原始文件被收集，然后根据文件类型分发到相应的处理器进行解析。解析过程中使用映射规则进行表头识别和数据标准化。处理结果添加数据来源标识后进入待合并队列。所有处理完成的结果通过pandas的concat操作合并为单一DataFrame，最后按照模板定义的列顺序调整后保存到输出文件。

### 表头识别逻辑

表头识别是数据处理的关键环节，采用多级匹配策略。首先进行精确匹配，尝试在配置中查找与原始表头完全相同的项。如果精确匹配失败，则进行关键词匹配，扫描原始表头中是否包含配置中定义的关键词。匹配成功后会建立映射关系，未能识别的列会保留原始列名或根据列位置进行推断。

### 资源管理策略

程序采用自适应的资源管理策略，根据系统配置和实时资源状态动态调整处理参数。初始化阶段检测CPU核心数、内存总量和可用内存量，根据检测结果生成基础资源配置方案。处理过程中实时监控资源使用情况，当CPU或内存使用率超过90%阈值时自动降速。降速策略包括减少并行线程数、增加批次间延迟等，确保系统稳定性。

## 扩展指南

### 添加新银行支持

要为新的银行添加支持，需要修改两个配置文件。在 `config.yaml` 的 `bank_bin_database` 部分添加新银行的BIN码段，在 `mappings.yaml` 的 `header_mapping` 部分添加新银行的表头映射规则。如果新银行有特殊的字段格式，还需要在 `field_types` 和相应的解析代码中添加支持。

### 自定义输出列

修改输出列结构有两种方式。方式一是编辑模板文件 `流水汇总模板.xlsx`，调整列的顺序和名称，程序会自动读取新的列结构。方式二是修改 `mappings.yaml` 中的 `fallback_expected_columns` 配置，定义硬编码的预期列列表。

### 性能调优

处理大量文件时可以采取以下优化措施：确保系统有足够的内存空间，建议可用内存大于待处理文件总大小的两倍；启用并行处理（默认启用）以充分利用多核CPU；对于超大型文件，考虑拆分为多个小文件处理；安装psutil包以启用更精确的资源监控。
